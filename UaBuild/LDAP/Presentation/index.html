Why are Tools So Great?

  • Enterprise Java Application implementation is not always a walk in the park.
  • Used when your materials don't match your powertools
  • Not always interactive (welcome automation)
  • Improve efficiency and productivity
  • Tools can/should make your project more agile

There is no turn-key solution for Enterprise Java Applications. There is always something that goes wrong during setup/configuration. Does anyone like woodworking? Not carpentry. I like to make furniture. Call it a passion. Some like robots. I like robots too, but I like woodworking more.

In woodworking, you have powertools. They make your life a lot easier. Enterprise Applications have their powertools tool. Peoplesoft, for example, has tons. Sometimes, they're not enough. For example, in woodworking, you can make a jig for certain tools for special cuts. The same goes for enterprise java applications. Sometimes your materials don't fit your powertools. A jig is all that's necessary to get the most out of your powertool.

The Right Tool for the Job

Development

  • CheckStyle (checkstyle.sourceforge.net)
  • Find Bugs (findbugs.sourceforge.net)
  • Mylyn (wiki.eclipse.org/index.php/Mylyn_FAQ)
  • Liquibase (liquibase.org)
  • Hudson CI (hudson-ci.org)

Mylyn

  • Eclipse plugin
  • Integrates IDE with Issue tracking
  • Separates work based on issue

Checkstyle

  • Integrates with Hudson CI
  • Checks code against style guidelines
  • Looks for redundant code
  • Extensible

Findbugs

  • Integrates with Hudson CI
  • Cross references code to find weaknesses in java
  • Looks for redundant/common code
  • Looks for smells to refactor

Testing

  • Groovy/JRuby(groovy.codehaus.org/jruby.org)
  • Hudson CI (hudson-ci.org)
  • Cobertura (cobertura.sourceforge.net)
  • Selenium seleniumhq.org
  • Amazon EC2 aws.amazon.com/ec2

Groovy/JRuby

  • Can be used to bootstrap sandbox environments for testing
  • AntBuilder!!
  • Scriptable access to the application
  • SQL is very difficult to test. Groovy can be used to execute/validate SQL quickly and easily

Cobertura

  • Code Coverage for tests
  • Builds confidence in tests
  • Open source
  • Integrates with Hudson CI

Selenium

  • For usability testing
  • Developers don't have to establish tests. BSA's or DME's can establish them.
  • Intuitive browser plugins to develop tests with
  • Cross browser testing capable
  • Clusterable to run several tests at once
  • Integrates with puppet an Hudson CI

Amazon EC2

  • Non reliant on Sysadmin or DBA
  • Can get a system quickly configured just the way you like
  • Toss it away when you're done
  • Don't need it to stay up. Just long enough to do tests.
  • Can be remotely configured with puppet or chef
  • Can be remotely triggered through Hudson CI to kick off tests

Database

  • Liquibase
  • Groovy/JRuby

Liquibase

  • Stores version within database instead of in a file
  • Can be used to rollback changes or push updates on schema and data changes by version
  • Supports anything under the sun
  • Actively maintained
  • Handles database versioning internally
  • Capable of rolling back versions
  • Uses internal DATABASECHANGELOG and DATABASECHANGELOGLOCK tables to manage versions
  • Described in XML, so parsing and translation is easy
  • Produces documention changes for each version

Configuration Management

  • Packaging
  • Liquibase
  • Groovy
  • Puppet or Chef
  • Sandboxes

Puppet and Chef

  • tools that can be used to manage configuration of an entire machine
  • sometimes referred to as sysadmin tools
  • can maintain packages on a machine pertaining to software configuration
  • can be used in a cluster configuration to distribute changes to different environments
  • scriptable management and monitoriing

Developing Tools

  • Invest in personal tools
  • Don't ever throw away a tool
  • Periodic problem solving discussion in development
  • Small quickly developed tools lead to larger maintained tools
  • Weigh maintenance cost
  • Use a sandbox before pushing tools on developers to establish parameters for training

Mosaic/KFS Implementation: A Use Case

Continuous Integration

  • Use Hudson CI to notify of build failures
  • RPM Packages built in CI
  • Use CI to handle build priorities to prevent failed builds from being packaged
  • CI manages integration and unit testing through the sandbox
  • Uses Amazon EC2 with selenium grid to verify usability testing in sandbox

Configuration Management

  • Currently using Hudson CI and ant tools to build rpm packages
  • Sandbox uses puppet to manage configuration changes
  • Sandbox uses puppet to create packages and distribute them
  • Sandbox uses puppet to stay up to date
  • Use liquibase to push database version updates to environments

Data Migration

  • Problems that occur with migrating legacy data
  • Problems that occur with migrating in development
  • Tools for migrating data
    • Liquibase
    • PL/SQL
    • Java/Groovy
  • Patterns using the aforementioned tools

UA Data Migration

FRS to RDBMS

  • FRS runs on a mainframe server platform
  • Communication protocols are (FTP, SFTP, and Telnet for terminal communication)
  • Requires DBA to import data with SQL Loader

RDBMS to RDBMS

  • Uses secondary schema to convert information
  • Mostly written with PL/SQL
    • Requires DBA to set install packages and set grants
    • Automation requires difficult to test shell scripts

Versioning Database Schemas

  • Migration scripts are always coupled with the software version
  • SVN handles versioning for us with tags and revision #'s
  • Jira integrates with SVN and can distribute database changes as part of a release
  • Most practices provide unidirectional versioning

Patterns for Liquibase

  • Store changes per version
  • Use the SVN revision number for changelog ids
  • One file per database object
  • Developers don't modify the database
  • RoR methodology in development
  • 2 scripts. One for updating, and for migrating

PL/SQL to Migrate Data

  • Difficult to test
  • May require DBA intervention
  • Special knowledge of database infrastructure is required
  • Native to RDBMS
  • Can give large performance gains
  • Migration process is required to be segmented

Java and Groovy for Migration

  • Easy to test
  • Can combine ETL in a process
  • Database agnostic
  • Data can be loaded natively with the application
  • Easily integrates with build processes

Development and Testing

  • Developers use liquibase to test database update changes
  • Use groovy to verify ETL tests
  • Cobertura is used to check code coverage
  • Checkstyle in use in sandbox
  • Findbugs in use in sandbox

Tool Integration

  • Configuration management change notifications sent through twitter and confluence update
  • Jira Issue tracking used to manage releases
  • Continuous Integration updates sent through email and twitter
